# 理论驱动的产品开发流程指引

## 当用户提出新功能想法时

你需要引导用户遵循"理论驱动的产品开发流程"：

### 第一步：识别当前阶段

判断用户处于哪个阶段：
1. **理论研究阶段：** 有想法但理论基础不清楚
2. **需求设计阶段：** 理论清楚，需要设计产品
3. **MVP 验证阶段：** 需求清楚，需要快速验证
4. **迭代优化阶段：** MVP 完成，需要优化
5. **产品化阶段：** 验证有效，需要正式开发

### 第二步：引导用户完成当前阶段

#### 如果是理论研究阶段：

提出以下问题：
1. 这个想法背后的理论支撑是什么？
2. 为什么这个问题重要？
3. 理论如何指导产品设计？

引导用户创建理论文档（`docs/理论基础/{理论名称}.md`）

#### 如果是需求设计阶段：

提出以下问题：
1. 如何将理论转化为产品功能？
2. 用户如何使用这个产品？
3. 如何验证产品是否有效？

引导用户创建需求文档（`docs/产品设计/{产品名称}-需求文档.md`）

#### 如果是 MVP 验证阶段：

帮助用户：
1. 选择合适的实现工具（Coze/Dify/n8n/代码）
2. 实现核心功能（仅核心）
3. 准备测试案例
4. 记录验证结果

#### 如果是迭代优化阶段：

帮助用户：
1. 分析 MVP 反馈
2. 确定优化优先级
3. 实施优化
4. 再次验证

#### 如果是产品化阶段：

帮助用户：
1. 完整功能实现
2. 工程化和质量保证
3. 文档完善
4. 部署上线

### 第三步：确保输出物

每个阶段都要有明确的输出：
- 理论研究 → 理论文档
- 需求设计 → 需求文档
- MVP 验证 → POC 原型 + 验证记录
- 迭代优化 → 优化版本 + 优化文档
- 产品化 → 生产级产品 + 完整文档

### 第四步：检查清单

使用对应阶段的检查清单，确保完成所有必要步骤。

## 核心原则

1. **理论先行：** 确保产品有坚实的理论基础
2. **快速验证：** 用 MVP 验证核心假设，避免过度投入
3. **文档驱动：** 每个阶段都要有清晰的文档
4. **迭代优化：** 根据反馈持续改进

## 灵活应用

- 简单功能可以简化流程
- 可以根据情况跳过或回退
- 重点是思考清楚，而不是机械执行
