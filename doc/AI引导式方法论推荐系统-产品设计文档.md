## 核心概念

### 1.1 什么是隐性知识与显性知识？

**隐性知识（Tacit Knowledge）**：
- 你知道，但说不出来
- 例如：骑自行车的平衡感、"这个设计不对劲"的直觉

**显性知识（Explicit Knowledge）**：
- 可以用语言清晰表达的知识
- 例如：数学公式、操作步骤

**本产品的核心价值**：
通过 AI 提问，帮助用户把**隐性知识转化为显性知识**

```
用户内心："我的项目有问题，但不知道怎么说..."（隐性）
    ↓ AI 提问引导
用户表达："原来是团队职责不清导致的！"（显性）
```

### 1.2 什么是信息熵减？

**信息熵（Entropy）**：不确定性的度量

```
初始状态：30 个方法论，完全不确定 → 熵最高
    ↓ 问题 1："是分析还是决策？"
中间状态：15 个方法论 → 熵降低 50%
    ↓ 问题 2："涉及多人吗？"
中间状态：8 个方法论 → 熵继续降低
    ↓ 问题 3："需要可视化吗？"
最终状态：3 个方法论 → 熵接近 0，确定性高
```

**每个好问题都应该显著降低熵**

### 1.3 什么是贝叶斯推理？

**贝叶斯更新**：根据新证据更新概率

```javascript
// 初始概率
P(鱼骨图) = 3.3%  // 30 个方法论之一

// 用户回答："我想分析多因素问题"
P(鱼骨图 | 多因素分析) = 25%  // 概率上升

// 用户回答："需要团队协作"
P(鱼骨图 | 多因素分析 + 团队协作) = 60%  // 继续上升

// 用户回答："希望可视化"
P(鱼骨图 | 所有条件) = 85%  // 高置信度，可以推荐
```

### 1.4 苏格拉底式提问法

**苏格拉底（Socrates）的教学方法**：
- 不直接给答案
- 通过提问引导学生自己发现答案
- 学生在回答过程中深化理解

**本产品采用相同理念**：
- AI 不直接推荐方法论
- 通过提问让用户自己"发现"答案
- 用户在过程中学习思维框架

### 1.5 邓宁-克鲁格效应（Dunning-Kruger Effect）

**什么是邓宁-克鲁格效应？**

这是一个认知偏差现象：能力欠缺的人往往高估自己的能力，而真正有能力的人反而会低估自己。

**四个认知阶段**：

```
自信度
  ↑
  │     ①愚昧山峰
  │      /\
  │     /  \
  │    /    \___②绝望之谷
  │   /          \
  │  /            \___③开悟之坡
  │ /                  \________④持续平稳高原
  │/
  └────────────────────────────────→ 知识/能力
  
不知道     知道      知道        知道
自己不知道  自己不知道  自己知道    自己知道
                              (且知道边界)
```

**① 愚昧山峰（Unconscious Incompetence）**
- **特征**：知识很少，但自信爆棚
- **心态**："我都懂" "这很简单" "不需要学"
- **问题**：不知道自己不知道
- **占比**：大多数初学者

**② 绝望之谷（Conscious Incompetence）**
- **特征**：开始学习，发现自己啥都不懂
- **心态**："太难了" "我学不会" "好复杂"
- **问题**：知道自己不知道，容易放弃
- **占比**：开始学习的人

**③ 开悟之坡（Conscious Competence）**
- **特征**：持续学习，能力稳步提升
- **心态**："我开始理解了" "原来是这样"
- **问题**：需要刻意练习
- **占比**：坚持学习的人

**④ 持续平稳高原（Unconscious Competence）**
- **特征**：知识丰富，自信适度且稳定
- **心态**："我知道，也知道我不知道什么"
- **问题**：需要持续精进
- **占比**：少数专家

**为什么这对产品设计很重要？**

你的产品要服务不同阶段的用户，每个阶段需要不同的策略：

| 阶段 | 用户心态 | 产品策略 | 设计重点 |
|------|---------|---------|---------|
| 愚昧山峰 | "我不需要" | 温和引导，不说教 | 降低门槛，游戏化 |
| 绝望之谷 | "太难了" | 快速见效，建立信心 | 简化流程，即时反馈 |
| 开悟之坡 | "我想学更多" | 提供深度内容 | 对比分析，进阶功能 |
| 平稳高原 | "我需要工具" | 高效工具，不啰嗦 | 快速访问，个性化 |

## 产品愿景

### 2.1 核心问题

**用户痛点**：
1. 不知道有哪些思维工具
2. 知道工具名称，但不知道何时用
3. 有问题，但不知道怎么描述
4. 搜索关键词都不知道用什么

**传统解决方案的问题**：
```
❌ 搜索引擎：需要知道关键词
❌ 方法论列表：看不懂，选不出
❌ 专家咨询：成本高，不及时
```

### 2.2 产品定位

**一句话描述**：
像 Akinator 猜人物一样，通过 AI 提问帮你找到最适合的思维工具

**核心价值**：
1. **零门槛**：不需要任何背景知识
2. **教育性**：边用边学，理解方法论
3. **精准性**：5-8 个问题找到答案
4. **趣味性**：像玩游戏一样有趣

### 2.3 用户旅程（基于邓宁-克鲁格效应）

#### 阶段 1：愚昧山峰 → 绝望之谷（第一次使用）

**用户状态**：
- 不知道有思维方法论这回事
- 觉得"凭感觉就能解决问题"
- 可能是被朋友推荐或偶然发现产品

**用户心声**：
> "我有个问题，但不知道怎么办..."
> "这些方法论是什么？我需要吗？"

**产品策略**：
- ✅ 不要一开始就讲理论（会被拒绝）
- ✅ 用简单的问题吸引进来："有问题？我来帮你理清思路"
- ✅ 通过 AI 提问引导（路径 1）
- ✅ 在回答过程中潜移默化地教育

**交互流程**：
```
欢迎页：
"🎯 有个问题不知道怎么解决？让我帮你理清思路"
[开始] ← 低门槛，不提"学习"

AI 提问：
"你是想分析问题，还是做决策？"
→ 用户："呃...分析问题吧"（开始意识到有区别）

AI 继续：
"是想找出根本原因（为什么会这样），还是评估现状（了解各方面情况）？"
→ 用户："哦！原来还有这个区别...找原因"（进入绝望之谷）

AI 引导：
"原因是单一的，还是多个因素互相影响？"
→ 用户："多个因素"（学到了"系统性思维"的概念）

推荐结果：
"推荐鱼骨图！因为它能帮你梳理多个因素..."
→ 用户："原来这就是鱼骨图！"（开始开悟）
```

**结果**：
- 找到了方法论
- 理解了基本概念
- 意识到"原来我不懂的还挺多"
- 愿意继续学习

#### 阶段 2：绝望之谷 → 开悟之坡（第 2-5 次使用）

**用户状态**：
- 知道有很多方法论，但记不住
- 知道自己需要学习，但觉得复杂
- 想快速找到答案，不想从头学

**用户心声**：

**产品策略**：
- ✅ 提供快速匹配（路径 2）
- ✅ 减少问题数量（3-5 个就够）
- ✅ 提供对比功能（A 和 B 有什么区别）
- ✅ 显示历史记录（你上次用的是...）

**交互流程**：
```
欢迎页（针对回访用户）：
"👋 欢迎回来！"
[继续上次的实践] [开始新的探索]

快速匹配：
用户输入："我想找出项目延期的原因"
↓
AI 理解意图：因果分析
↓
AI 追问 2-3 个关键问题：
"是单一原因还是多因素？"
"需要团队一起分析吗？"
↓

"鱼骨图适合多因素，5 Why 适合单一原因"
```

**结果**：
- 更快找到答案（2-3 分钟）
- 开始理解不同方法的区别
- 建立了信心："我开始懂了"
- 愿意尝试更多方法

成长记录：

## 技术原理

### 3.1 Akinator 的工作原理

**Akinator 如何猜对人物？**

1. **大数据库**：存储了数万个人物及其特征
2. **决策树**：每个问题都能排除一半选项
3. **概率计算**：实时更新每个人物的可能性
4. **智能提问**：选择最有区分度的问题

**示例**：
```
数据库：10000 个人物

问题 1："是真实人物吗？"
→ 回答"是" → 剩余 6000 个

问题 2："是政治人物吗？"
→ 回答"否" → 剩余 4000 个

问题 3："是演员吗？"
→ 回答"是" → 剩余 1500 个

...

问题 15："是漫威电影演员吗？"
→ 回答"是" → 剩余 3 个
→ 猜测："是小罗伯特·唐尼吗？"
```

### 3.2 本产品的工作原理

**核心机制**：

```
方法论数据库：30 个方法论 + 特征标签
    ↓
AI 问题生成器：根据上下文生成最优问题
    ↓
用户回答：选择选项
    ↓
候选集缩小：排除不匹配的方法论
    ↓
重复 5-8 次
    ↓
推荐 Top 3 方法论 + 理由
```

### 3.3 AI 的角色

**AI 不是用来"创造"，而是用来"引导"**

| 任务 | 是否需要 AI | 说明 |
|------|-----------|------|
| 存储方法论 | ❌ | 用数据库 |
| 生成问题 | ✅ | AI 根据上下文动态生成 |
| 理解用户回答 | ✅ | AI 解析选择的含义 |
| 计算匹配度 | ✅ | AI 推理哪些方法论最合适 |
| 解释推荐理由 | ✅ | AI 用自然语言说明 |

**为什么用 AI 而不是固定决策树？**

固定决策树：
```
if (answer1 == "分析") {
  if (answer2 == "多人") {
    recommend("鱼骨图");
  }
}
```
❌ 问题：僵硬、无法适应、难以维护

AI 动态生成：
```
AI 看到：用户说"分析"，还说"多人"
AI 推理：可能需要协作工具
AI 生成下一个问题："需要可视化吗？"
```
✅ 优势：灵活、可学习、自然对话

### 4.2 问题设计框架

**问题层级**：

```
Level 1 - 大方向（必问）
目的：排除 50% 选项
示例：
- "你是想分析问题，还是做决策，还是激发创意？"
- "这是个人思考，还是团队协作？"

Level 2 - 场景特征（必问）
目的：继续缩小到 20%
示例：
- "问题是简单的，还是复杂的？"
- "时间充裕，还是紧急？"

Level 3 - 具体需求（按需问）
目的：精确匹配
示例：
- "需要可视化图表吗？"
- "需要量化数据吗？"

Level 4 - 确认细节（可选）
目的：最后确认
示例：
- "希望结果是结构化的吗？"
- "需要多轮迭代吗？"
```

**好问题的标准**：

✅ **互斥性**：选项之间不重叠
```
好：○ 个人 ○ 小团队(3-10人) ○ 大团队(10+人)
坏：○ 个人 ○ 团队 ○ 很多人  // "团队"和"很多人"重叠
```

✅ **完备性**：覆盖所有可能
```
好：○ 分析 ○ 决策 ○ 创新 ○ 其他
坏：○ 分析 ○ 决策  // 缺少其他情况
```

✅ **易理解**：用户能快速判断
```
好：○ 找出根本原因（为什么会这样）
坏：○ 进行因果分析  // 术语太专业
```

✅ **高区分度**：能显著缩小范围
```
好：问"是分析还是决策？" → 排除 50%
坏：问"需要纸笔吗？" → 几乎不排除
```

### 4.3 选项设计原则

**在选项中嵌入知识**：

❌ 普通选项：
```
○ 是
○ 否
```

✅ 教育性选项：
```
○ 是，我想找出根本原因（深挖为什么）
○ 否，我只需要了解现状（知道是什么）
```

**示例对比**：

| 问题 | 普通选项 | 教育性选项 |
|------|---------|-----------|
| 涉及多少人？ | ○ 少 ○ 多 | ○ 个人思考<br>○ 小团队协作(3-10人)<br>○ 大团队共识(10+人) |
| 时间紧急吗？ | ○ 是 ○ 否 | ○ 很紧急(1-2天内)<br>○ 适中(1周左右)<br>○ 充裕(1个月以上) |
| 需要可视化吗？ | ○ 需要 ○ 不需要 | ○ 需要图表(便于理解和沟通)<br>○ 文字即可(重点是思考过程) |

export function buildUserPrompt(
  conversationHistory: ConversationMessage[],
  allMethodologies: Methodology[]

  return `
# 对话历史
${conversationHistory.map((msg, i) => {
  if (msg.role === 'ai') {
    return `AI 问题 ${i + 1}: ${msg.question}`;
  } else {
    return `用户回答: ${msg.selectedOption}`;
  }
}).join('\n')}

# 当前状态
- 已提问数：${conversationHistory.filter(m => m.role === 'ai').length}
- 剩余候选方法论：${getCurrentCandidates(conversationHistory).length} 个

请生成下一个问题，或者给出推荐（如果已经足够确定）。
`;
}

### 6.3 核心 API 实现

```typescript
// src/api/ai-conversation.ts

import OpenAI from 'openai';
import { SYSTEM_PROMPT, buildUserPrompt } from '@/lib/ai-prompts';
import { getAllMethodologies } from '@/lib/methodology-data';

const openai = new OpenAI({
  apiKey: process.env.NEXT_PUBLIC_OPENAI_API_KEY,
});

export async function continueConversation(
  conversationHistory: ConversationMessage[]
): Promise<ConversationMessage> {
  
  const methodologies = getAllMethodologies();
  
  const response = await openai.chat.completions.create({
    model: "gpt-4",
    messages: [
      {
        role: "system",
        content: SYSTEM_PROMPT
      },
      {
        role: "user",
        content: buildUserPrompt(conversationHistory, methodologies)
      }
    ],
    temperature: 0.7,
    response_format: { type: "json_object" }
  });
  
  const content = response.choices[0].message.content;
  const aiResponse = JSON.parse(content);
  
  return {
    id: generateId(),
    role: 'ai',
    content: aiResponse.question || '推荐结果',
    timestamp: Date.now(),
    question: aiResponse.question,
    options: aiResponse.options,
    explanation: aiResponse.explanation,
  };
}
```

### 7.2 优化指标

- 准确率 > 75%
